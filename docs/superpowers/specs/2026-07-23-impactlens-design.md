# ImpactLens — Hackathon MVP Design

**Date:** 2026-07-23
**Status:** Revised after scope review
**Track:** Yunus Social Innovation — AI-Powered Impact Intelligence
**Build target:** Working local demo by 20:00

## 1. Product summary

ImpactLens turns previously unseen tabular programme data into an explainable first-pass impact dashboard.

The core demonstration is:

> **Previously unseen spreadsheet → inferred meaning → proposed KPIs → deterministic calculations → source evidence.**

Claude interprets what uploaded tables and fields mean. Deterministic application code validates and calculates the metrics. An evidence layer traces every result back to its source. This separation is the product's main innovation and must survive every scope cut.

ImpactLens is a best-effort analysis tool, not a universal spreadsheet engine. The MVP works best when a table contains one record per row, recognisable headers, and at least one countable, numeric, categorical, date, stage, or target field.

The application has one shared project view for programme managers and participating enterprises. Authentication, permissions, and role-specific views are deferred.

## 2. Hackathon MVP scope

ImpactLens accepts previously unseen CSV or XLSX files and creates an explainable first-pass dashboard. It deterministically parses and profiles each table, then runs one bounded Claude Code print-mode analysis to infer field roles and propose up to four KPIs from a constrained formula language.

The user reviews and accepts or removes proposed KPIs. Accepted values and chart series are calculated locally rather than generated by the model. Each KPI includes an evidence view containing its source file, worksheet, fields, formula, coverage, confidence, assumptions, and exclusions.

Tables are analysed independently by default. Cross-file metrics are enabled only when one exact, deterministic key relationship passes strict type, uniqueness, coverage, and collision checks and the user confirms it. The MVP does not perform fuzzy entity resolution or arbitrary multi-table modelling.

The generated dashboard contains an Overall view, basic data-quality warnings, and an Outlook area that defaults honestly to **Insufficient evidence**. A directional Outlook is a stretch feature. Natural-language additional metrics, advanced project-risk detection, forecasting, fuzzy joins, and a materialised row-level canonical observation model are outside the MVP.

The demo proves the same generic pipeline using a locally seeded Aurelia example and one previously unseen live upload.

## 3. Goals and non-goals

### MVP goals

- Accept any number of CSV/XLSX files within the combined size and parsed-row safeguards.
- Profile tables and fields deterministically before any model call.
- Use one `claude -p` run to return a validated semantic plan and three or four proposed KPIs.
- Reject model proposals that reference missing fields or unsupported calculations.
- Let the user accept or remove proposed KPIs.
- Calculate accepted KPI values and chart series locally.
- Generate one useful Overall dashboard with evidence for every KPI.
- Surface a small set of defensible data-quality warnings.
- Prevent ambiguous joins from producing cross-file metrics.
- Open a locally seeded Aurelia project immediately during the pitch.
- Demonstrate one unseen live upload without dataset-specific mappings.
- Show a useful limited-result state when the data cannot support meaningful analysis.
- Relate proposed KPIs to relevant impact frameworks without claiming certification or formal compliance.

### Explicit non-goals

- Authentication, accounts, permissions, or separate manager and enterprise views.
- PDFs, images, free-form documents, APIs, or live database connectors.
- Password-protected or heavily formatted spreadsheets.
- Fuzzy entity resolution, composite joins, many-to-many joins, or a manual join editor.
- A row-level entity/observation database or general analytical warehouse.
- Arbitrarily nested formulas, AI-generated SQL, AI-generated code, or AI-generated chart components.
- Drag-and-drop dashboard composition.
- Natural-language creation of an additional KPI in the MVP.
- Numerical success probabilities or forecasting.
- Generic claims about viability, underserved groups, representation, or causal impact.
- Formal ESG, SDG, IRIS+, or other standards compliance and assurance.
- A bundled external standards catalog or automatic invention of official indicator IDs.
- Production queues, collaboration, export, object storage, or deployment hardening.

### Framework-aware IMM lens

ImpactLens follows a compact impact measurement and management cycle:

1. **Set goals and intentions** — capture what the project is trying to achieve and what the analysis should pay attention to.
2. **Theory of Change** — identify available evidence of activities → outputs → near-term outcomes → longer-term impact. The MVP does not build or validate a full Theory of Change; it reports which links appear in the supplied context and data and which are missing.
3. **Select indicators** — propose calculable quantitative or count-based indicators and identify relevant qualitative evidence fields. Free-text evidence may inform the interpretation and framework rationale, but qualitative theme synthesis is deferred from the MVP.
4. **Manage and adapt** — use results, evidence gaps, and warnings to support operational learning and follow-up decisions.

The semantic interpretation may apply these lenses when relevant:

- **Five Dimensions of Impact:** What, Who, How Much, Contribution, and Risk. This is the primary impact-specific lens. The review shows each dimension as **identified**, **partial**, or **not found**, with supporting fields or project context. "Not found" means the upload lacks evidence; it is not a negative impact rating.
- **Sustainable Development Goals:** candidate alignment to one or more of the 17 goals. A high-level thematic match is not an official SDG indicator measurement. The MVP must not invent an SDG target or indicator ID.
- **IRIS+:** candidate metric-search terms or a possible conceptual match. An exact IRIS+ metric code is shown only when the user supplied it in project context; the MVP does not contain the full IRIS+ catalog.
- **ESG:** optional Environmental, Social, or Governance classification where the project context and fields support it. ESG is treated as a broad reporting lens, not a single certification standard.
- **Triple Bottom Line:** optional People, Planet, and economic-value lens. It is a communication aid, not a formal metric standard.

Framework suggestions are semantic metadata returned within the existing Claude Code run. They never change a formula, supply a missing denominator, create a benchmark, or make an unsupported metric calculable.

Authoritative reference points for the terminology are [Impact Frontiers' Five Dimensions of Impact](https://impactfrontiers.org/norms/five-dimensions-of-impact/), the [IRIS+ introduction to IMM](https://iris.thegiin.org/introduction/), the [IRIS+ metric catalog](https://iris.thegiin.org/metrics/), the [UN Sustainable Development Goals](https://sdgs.un.org/goals), and the [official UN SDG indicator framework](https://unstats.un.org/sdgs/indicators/indicators-list/).

## 4. Supported data boundary

One project accepts:

- any number of CSV or XLSX files;
- up to 10 MB combined;
- up to 25,000 parsed rows across all tables;
- multiple XLSX worksheets;
- UTF-8 CSV, including byte-order marks;
- comma, semicolon, or tab delimiters; and
- quoted multiline CSV cells.

The preferred input shape is one record per row with a single header row. The parser makes a best effort with sparse tables, mixed types, and unusual number formats, but the MVP does not promise correct interpretation of:

- merged cells or several header rows;
- formulas without cached results;
- pivot-table or presentation-style output;
- tables containing several unrelated grains;
- password-protected workbooks;
- primarily free-text material with no measurable fields; or
- ambiguous dates, decimals, currencies, or percentages that cannot be resolved safely.

Unsupported or ambiguous structures create visible warnings. They are not silently coerced.

## 5. User experience

### 5.1 Projects

The landing page lists locally persisted generated projects and offers **Create project**. Two examples protect the live pitch:

- **Aurelia Propel**, produced locally from all supplied raw files by the generic seed command; and
- a small committed synthetic fallback project containing no restricted data.

Each card shows project name, number of sources, accepted KPIs, status, and last generated time.

### 5.2 Create and upload

The project form asks only:

1. **Project name**
2. **What is this project trying to achieve?**
3. **Anything the analysis should pay attention to?** — optional

The optional field may name a Theory of Change, SDG, IRIS+ metric, ESG priority, impact dimension, or Triple Bottom Line focus. No separate framework questionnaire is added to the live flow.

The user uploads CSV/XLSX files. The UI shows file names, sizes, the combined limit, and clear validation errors. There is no file-count limit.

Real processing progress uses four stages:

1. Reading files
2. Profiling tables
3. Interpreting the data
4. Preparing the review

### 5.3 Review

The review screen is decision-focused rather than a schema inspector:

1. **What we understood** — a short summary of table purposes, important fields, ambiguities, and limitations.
2. **Files and detected tables** — file/sheet, rows, fields, and parse warnings.
3. **Impact framework lens** — compact Five Dimensions coverage and optional candidate SDG, IRIS+, ESG, or Triple Bottom Line tags, shown only when supported.
4. **Proposed KPIs** — three or four cards with accept/remove controls.
5. **Potential relationship** — shown only when one exact safe join candidate exists, with a confirmation control.
6. **Advanced details** — collapsed profiles, confidence, and model output for inspection.

Each KPI card shows:

- name and plain-language definition;
- formula summary;
- source file/sheet and fields;
- coverage estimate;
- confidence;
- why it was proposed; and
- assumptions or exclusions; and
- optional framework tags with a short evidence-based rationale.

Framework tags are visually secondary to the KPI definition and evidence. The UI labels them **Candidate alignment**, never **Compliant**, **Certified**, or **Official metric**.

Unsupported proposals are removed by deterministic validation before the user sees them. If fewer than two valid KPIs remain, the review explains what evidence is missing rather than padding the list with weak suggestions.

### 5.4 Dashboard

The project dashboard retains the three-part product concept without making all three parts equal implementation dependencies:

- **Overall** — the official MVP and primary pitch view.
- **Early Warnings** — basic data-quality warnings in the MVP; limited project warnings are stretch.
- **Outlook** — **Insufficient evidence** in the base MVP; a transparent directional heuristic is stretch.

Every KPI has **View evidence**, opening a drawer with its source, fields, formula, coverage, missing/excluded counts, confidence, assumptions, and up to five example source row numbers.

## 6. Runtime pipeline

### Stage 1: Parse and profile

Each CSV file or XLSX worksheet becomes an in-memory parsed table with stable source and field identifiers, original row numbers, raw values, and parser warnings.

The deterministic profiler produces:

```ts
type SourceProfile = {
  sourceId: string;
  filename: string;
  sheetName?: string;
  rowCount: number;
  fields: FieldProfile[];
  warnings: DataWarning[];
};
```

Each field profile contains its original header, inferred physical type, missingness, uniqueness, numeric/date range when valid, limited representative samples, and inference confidence. Email and phone-like samples are redacted.

### Stage 2: Interpret once with Claude Code

The application creates an isolated per-run directory containing `analysis-input.json`, which includes:

- the two project-context fields;
- compact table and field profiles;
- redacted representative samples; and
- the exact structured-response contract.

It does not place raw uploads or repository files in the analysis workspace. Claude Code receives the benefits of its agent harness and built-in Read tool while being limited to the redacted analysis packet.

The server launches Claude Code as a shell-free subprocess equivalent to:

```powershell
claude -p "Read analysis-input.json and return the requested semantic plan." `
  --output-format json `
  --json-schema "<semantic-plan-schema>" `
  --max-turns 3 `
  --max-budget-usd "<configured-budget>" `
  --tools Read `
  --permission-mode dontAsk `
  --safe-mode `
  --no-session-persistence
```

The subprocess working directory is the isolated run directory. It is invoked with an argument array and `shell: false`; no uploaded value is interpolated into a command string. A timeout terminates the run and produces a retryable interpretation error.

Claude returns one object containing both semantic interpretation and proposed KPIs:

```ts
type SemanticPlan = {
  tablePurposes: TablePurpose[];
  fieldRoles: FieldRole[];
  candidateTimeFields: FieldRef[];
  candidateDimensions: FieldRef[];
  candidateJoin?: CandidateJoin;
  immAssessment: {
    theoryOfChangeEvidence: Array<{
      stage: "activity" | "output" | "outcome" | "impact";
      status: "identified" | "partial" | "not_found";
      fieldRefs: FieldRef[];
      rationale: string;
    }>;
    impactDimensions: Array<{
      dimension: "what" | "who" | "how_much" | "contribution" | "risk";
      status: "identified" | "partial" | "not_found";
      fieldRefs: FieldRef[];
      rationale: string;
    }>;
    candidateFrameworks: FrameworkTag[];
  };
  proposedMetrics: MetricDefinition[];
  recommendedChart?: ChartRecommendation;
  ambiguities: string[];
};
```

Claude Code validates its structured output against `--json-schema`. The application then parses the JSON wrapper and validates the structured result again with Zod. A second deterministic validation rejects nonexistent field references, unsupported formulas, incompatible field types, unsafe joins, and invalid chart inputs.

### Stage 3: Review

The user accepts or removes metrics. When a safe exact join candidate exists, it remains disabled until the user explicitly confirms it.

### Stage 4: Calculate

The server reparses or uses the current in-memory tables and evaluates accepted definitions directly against their rows. It calculates metric values, coverage, exclusions, grouped chart series, and evidence metadata. Claude does not calculate values.

### Stage 5: Render and cache

The server stores calculated results, chart series, evidence metadata, and warnings as JSON. The dashboard renders from this cache, so opening an existing project does not rerun Claude or expose raw rows.

## 7. Constrained KPI language

Claude proposes three or four metrics. Each metric uses one source table unless the user has confirmed a safe exact join.

The MVP supports five calculation shapes:

1. `count`
2. `distinct_count`
3. `sum`
4. `average`
5. `ratio`

`count`, `distinct_count`, `sum`, and `average` are atomic expressions over one optional field and a small set of equality or non-empty filters. A `ratio` contains exactly two atomic expressions. Formula nesting beyond that is invalid.

`period_change` is implemented only if the core pipeline is stable and the plan contains one validated ordered time field. `stage_conversion` is a later stretch feature.

A metric definition contains:

```ts
type MetricDefinition = {
  id: string;
  name: string;
  definition: string;
  operation: "count" | "distinct_count" | "sum" | "average" | "ratio";
  sourceId: string;
  field?: FieldRef;
  numerator?: AtomicExpression;
  denominator?: AtomicExpression;
  filters: MetricFilter[];
  groupBy?: FieldRef;
  unit: string;
  confidence: number;
  rationale: string;
  assumptions: string[];
  frameworkTags: FrameworkTag[];
};
```

`FrameworkTag` contains a framework name, human-readable candidate alignment, confidence, evidence basis, rationale, and caveat. Allowed framework names are `five_dimensions`, `sdg`, `iris_plus`, `esg`, and `triple_bottom_line`. In the MVP, an SDG tag may name a goal number but not an official target/indicator; an IRIS+ tag may contain search terms but not a metric code unless that code came from user-supplied context.

The evaluator verifies field existence, physical-type compatibility, filters, grouping, and non-zero denominators. Missing values are excluded and counted; they are never replaced with zero.

## 8. Exact cross-file join rule

Tables are independent by default. The system exposes at most one candidate relationship in the MVP.

A cross-file relationship is eligible only when:

- the two key fields have compatible physical types;
- values match after exact deterministic normalisation only;
- match coverage is at least 90% on the relevant side;
- the required one-side key contains no duplicates;
- the audit finds no many-to-many relationship;
- no fuzzy or semantic matching is required;
- no composite key is required; and
- the user explicitly confirms the relationship on the review screen.

Normalisation may trim whitespace, normalise Unicode, and compare case-insensitively. It must not remove arbitrary words or guess that different names represent the same entity.

If any condition fails, cross-file metrics are rejected and each table is analysed independently. The review explains why the join was unsafe.

## 9. Dashboard content

### 9.1 Overall — core MVP

Overall contains:

- two to four accepted KPI cards;
- one primary line, bar, or funnel chart when the validated data shape supports it;
- a concise deterministic assessment assembled from calculated metric names, values, coverage, and warnings;
- visible data coverage; and
- evidence access for every KPI.

When supported, a compact **Impact evidence coverage** card shows the Five Dimensions as identified, partial, or not found. Candidate SDG, IRIS+, ESG, or Triple Bottom Line tags may appear beneath the relevant KPI or assessment. These are interpretive aids and do not count as additional calculated KPIs.

The chart selector is deterministic:

- ordered date/time grouping → line chart;
- categorical grouping → bar chart;
- explicitly ordered stage fields in one table → funnel chart;
- otherwise KPI cards only.

Empty or unsupported charts are omitted. Claude returns a recommendation, not executable visualisation code.

### 9.2 Early Warnings — limited MVP

The MVP implements only these data warnings:

- failed or partially failed parsing;
- high missingness;
- duplicate candidate identifiers;
- mixed or incompatible physical types;
- invalid dates or numbers;
- low comparison coverage; and
- an uncertain or rejected join.

If time permits, project warnings are limited to:

- an accepted directional KPI declined materially;
- a validated conversion declined materially; and
- a numeric target was missed.

The MVP does not infer low runway, viability, underrepresentation, underserved groups, widening disparities, sophisticated outliers, or unexpected event sequences without the benchmarks and semantics needed to support those claims.

### 9.3 Outlook — stretch

The base MVP renders **Insufficient evidence** and states which time, target, or directional evidence is missing.

If implemented after the core path is stable, Outlook uses four non-probabilistic states:

- **Improving**
- **Mixed**
- **Declining**
- **Insufficient evidence**

It is a transparent summary of at least two accepted directional KPIs with comparable periods or targets. The UI shows every contributing signal. It does not present a numeric score, success probability, forecast, or universal threshold as validated truth.

## 10. Persisted model

SQLite stores only five logical record types:

### `projects`

Project name, goal, optional analysis note, status, timestamps, and whether the project is synthetic.

### `sources`

Project ID, original filename, worksheet/table name, raw-file reference, parser status, and the table/field profile JSON.

### `analysis_runs`

Project ID, Claude model, prompt/schema version, validated semantic-plan JSON, status, safe error, and timestamps.

### `metrics`

Project ID, proposed definition JSON, accepted state, calculated result JSON, chart-series JSON, and evidence JSON.

### `findings`

Project ID, warning type, severity, explanation, action, and evidence JSON.

Raw uploads remain unchanged on ignored local disk. The application does not persist one database row for every source cell, entity, or observation. During generation it calculates directly from parsed source rows, then renders from cached aggregate results.

Evidence metadata stores:

- source filename and worksheet;
- source field identifiers;
- formula and filters;
- records used versus available;
- missing and excluded record counts;
- assumptions and confidence; and
- up to five example source row numbers.

## 11. Seed and fallback strategy

The supplied Aurelia dataset must not be committed or redistributed.

A local command receives the Aurelia dataset directory:

```powershell
bun run seed:aurelia -- "D:\Claude_Impact_Hackathon\Aurelia_Propel_IMM Dataset"
```

runs the same generic parse → profile → interpret → validate → calculate pipeline against all supported raw files in the locally supplied directory and creates the ignored SQLite state before the pitch. Application code contains no YSI-specific filenames, headers, mappings, formulas, or join rules.

Neither raw Aurelia files nor generated row-level data are committed. Cached local aggregate results remain ignored with the database.

The repository also contains a small synthetic fallback project fixture and seed command. It contains only invented data and produces a ready dashboard without requiring the restricted dataset. This protects the pitch if local Aurelia state is removed or the API is unavailable.

### Aurelia measurement themes the generic pipeline should discover

The Aurelia files contain four recurring measurement themes. They are validation targets for the locally seeded demo, not YSI-specific schemas, prompts, filename rules, or metric mappings in application code. The pipeline may propose them only when the uploaded fields and project context support the interpretation.

#### 1. Five-level Impact Funnel — YSI's core IMM metric

The signature framework appears in `AP1and2_April_25`, `AP3_April_2025_Midline`, `AP4_May_2026_Midline`, and `AP_April_2025_extra_data`:

```text
1. Inform     → people who received education or information
2. Engage     → people directly engaged with the product or service
3. Outcomes   → people who experienced a measurable outcome
4. Impact     → people who experienced deep or lasting impact
5. Societal   → people who experienced broader societal impact
```

Each stage also contains evidence about vulnerable or marginalised groups, women's share, how people were reached, and how the number was calculated. A representative pseudonymised row reports `7,500 → 6,450 → 5,600 → 5,600 → 3,500`.

When Claude Code identifies fields as an ordered reach-to-impact funnel within one table, deterministic code may render the five stages and check the expected non-increasing relationship. A later stage exceeding an earlier stage creates a data-quality warning, not an automatic correction: indirect reach, different reporting windows, or number-format ambiguity may explain the result.

#### 2. Organisational performance and financials

Recurring measures include revenue, EBITDA, net profit, MRR, runway, FTE and contract headcount, customer counts, equity/debt/grant funding, current funding requirement, and valuation. These fields are especially prominent in `AP1_April_23_Org_Performance_SA` and the AP1 endline.

Formats such as `R26 766 051`, `R0`, `$60500`, `4686.41`, `120.000`, and `$5,000` must not be aggregated blindly. The MVP may calculate a financial KPI only when currency and number interpretation are unambiguous and values are comparable. Otherwise it reports the field and format inconsistency as evidence or a warning.

#### 3. Knowledge and Behaviour self-assessment — capability lift

The programme measures founder capability across competencies such as Systems, Purpose, IMM, Customer, Design Thinking, Product-Service, Business Strategy, and Unit Economics. Baseline and later-wave files use Knowledge-versus-Behaviour and before/after Likert-style ratings.

Capability lift is calculable only when the semantic plan identifies comparable competency fields, an ordered wave/time field, and—when multiple files are involved—a confirmed exact relationship. Question drift or uncertain joins must prevent a claimed pre/post result. The review may still identify capability measurement as a theme and explain why a longitudinal KPI is unsupported.

#### 4. Programme feedback and satisfaction

Recurring evidence includes NPS-style likelihood-to-recommend scores, module ratings, satisfaction, biggest take-away, suggested improvements, top needs, and programme attribution. Numeric ratings can support an average or distribution when their scales are clear.

Free-text responses are recognised as qualitative evidence and may support framework rationale, but cross-response thematic summarisation is outside the MVP's single interpretation run. The dashboard should identify the available qualitative evidence and position synthesis as a production extension rather than silently ignoring it.

For the live seed, success means the generic semantic plan recognises the five-level funnel plus at least one of organisational performance, capability lift, or programme feedback. It does not need to calculate all four themes. Every calculated theme remains subject to the same formula, coverage, join, and evidence rules as an unseen upload.

## 12. Technical architecture

### Stack

- Next.js App Router with TypeScript
- Bun for packages and scripts
- Tailwind CSS with a small local component layer
- SQLite through Bun's built-in `bun:sqlite` driver
- Papa Parse for CSV
- SheetJS for XLSX
- Claude Code CLI in non-interactive `-p` mode
- Zod for every AI and API contract
- Recharts for line, bar, and funnel visualisations
- Vitest for the five automated MVP tests

The application lives in a new `impact-lens/` directory. It does not import, copy, extend, or depend on `ysi-dashboard/`.

### Routes

- `/` — project list
- `/projects/new` — one wizard containing context, upload, processing, and review
- `/projects/[id]` — generated project dashboard

### Server responsibilities

Next.js Node-runtime route handlers or server actions perform upload validation, raw-file storage, parsing, profiling, the single bounded Claude Code subprocess run, response validation, exact-join audit, KPI evaluation, warning generation, caching, and loading.

Claude Code is resolved from `CLAUDE_PATH`, then the process `PATH`, then the platform-native default under the user's home directory: `.local/bin/claude.exe` on Windows or `.local/bin/claude` on macOS. Startup health checks run the resolved binary with `--version`. Configuration uses `CLAUDE_MODEL`, `CLAUDE_MAX_BUDGET_USD`, and `CLAUDE_TIMEOUT_MS`; authentication remains owned by the locally installed and configured Claude Code CLI.

### Windows and macOS portability

- Application and seed scripts are TypeScript commands invoked through Bun; they do not depend on PowerShell, Bash, `grep`, or other shell-specific utilities.
- Paths use `node:path`, `node:os`, and absolute resolved paths. No drive letter, slash direction, home-directory spelling, or executable suffix is assumed in application code.
- Claude Code, seed, and parser subprocesses use argument arrays with `shell: false` on both platforms.
- Temporary and data directories use `path.join(process.cwd(), ".data", ...)` and are created recursively.
- Timeout cleanup first requests process termination and then force-kills after a short grace period using Node's cross-platform child-process API.
- Uploaded filenames are treated as labels only; generated random storage names prevent path traversal and Windows/macOS reserved-name collisions.
- CSV decoding and parsing normalise UTF-8 BOM and CRLF/LF differences.
- SQLite access uses the same schema on both platforms; database paths are configurable for tests and local operation.
- The README documents native Claude Code installation/authentication and a `CLAUDE_PATH` override separately for Windows and macOS.

### Client responsibilities

Client components manage the wizard, upload state, KPI accept/remove controls, optional safe-join confirmation, chart rendering, dashboard tabs, and evidence drawers. They do not calculate business metrics.

## 13. Trust, privacy, and failure behaviour

- Raw files and SQLite state live under ignored local application data.
- Claude receives compact profiles and limited redacted samples, not full uploads by default.
- The UI states that representative values are sent for interpretation.
- Every AI response is schema-validated and field-reference-validated.
- Framework tags require an evidence basis and rationale; unsupported or over-specific tags are removed during validation.
- Claude cannot execute code, SQL, formulas, or chart configuration.
- Missing values remain missing.
- Ambiguous dates, numbers, currencies, percentages, and joins create warnings rather than silent guesses.
- A failed source is isolated; successfully parsed tables may proceed.
- If no source succeeds, the project stops with a recoverable file error.
- If Claude Code is missing, unauthenticated, times out, exceeds budget, exits non-zero, or returns invalid output, parsed profiles remain intact and interpretation is retryable with a specific safe error.
- If valid KPIs cannot be proposed, the limited-result state explains which data shapes are missing.
- If a join is ambiguous, it cannot produce a cross-file metric.
- Cached dashboards open without rerunning Claude.
- The Aurelia example is labelled pseudonymised hackathon data and never presented as evidence about named real enterprises.

## 14. Functionality-first build order

Build and verify in this order:

1. CSV parser and deterministic profiler.
2. XLSX parser with multiple worksheets.
3. Claude semantic-plan schema and one validated response.
4. Five-operation formula validator and evaluator.
5. One synthetic fixture through parse → interpret → validate → calculate in an API/integration test.
6. Single wizard for upload → review → generated Overall dashboard.
7. Evidence drawer.
8. Persist generated project results.
9. Local Aurelia seed and committed synthetic fallback seed.
10. Basic data-quality warnings.
11. Visual polish and demo rehearsal.

The semantic-plan task includes IMM and framework metadata because it is returned by the same Claude Code run. Do not add a standards database, a second model run, or a framework configuration screen to the MVP.

Only after this path is stable, attempt stretch work in this order:

1. exact-key cross-file join;
2. directional Outlook;
3. natural-language extra KPI;
4. limited project warnings;
5. `period_change`, stage conversion, or additional charts.

## 15. Minimal verification plan

Automate exactly these five MVP tests first:

1. A CSV fixture is parsed and profiled correctly, preserving missing values.
2. An XLSX fixture with two worksheets is parsed into two tables.
3. Invalid Claude field references and unsupported formulas are rejected.
4. Supported KPI formulas return expected values, coverage, evidence, and safe divide-by-zero behaviour.
5. One end-to-end synthetic fixture reaches a dashboard cache with valid KPI evidence.

The third test also rejects an official SDG indicator ID or IRIS+ metric code that was not present in user-supplied context.

Manually smoke-test:

- local Aurelia seed;
- one unseen CSV;
- one unseen XLSX;
- Claude Code missing/authentication/timeout failure and retry;
- a dataset with no useful numeric, categorical, stage, target, or time fields;
- an ambiguous join; and
- refresh and project reload.

Before the pitch, run the automated tests, lint, production build, both seed commands, and two complete rehearsals.

## 16. Live demo script

1. Open the locally seeded Aurelia project.
2. State the problem: impact teams spend days reconciling inconsistent spreadsheets before analysis begins.
3. Show two or three calculated KPIs and one chart.
4. Open a KPI's evidence drawer and show source, formula, coverage, and exclusions.
5. Show the Five Dimensions coverage and one carefully labelled candidate framework alignment.
6. Show one basic data-quality warning.
7. Return to Projects and create a new project.
8. Enter the short goal and upload one small unseen CSV or XLSX file.
9. Let the real profiler and single bounded `claude -p` interpretation run.
10. Review the proposed KPIs and optional framework tags, remove one KPI, and generate the dashboard.
11. Open evidence for one live-generated KPI.
12. Close with the production roadmap: larger inputs, reviewed joins, reusable semantic layers, standards catalogs, scheduled monitoring, and richer Outlook.

Keep one known-good unseen CSV and one known-good unseen XLSX available. The live file should be small enough that interpretation completes within the allocated pitch segment.

## 17. MVP acceptance criteria

The MVP is complete when:

- one unseen CSV and one unseen XLSX can each reach review;
- Claude proposes at least two calculable KPIs when the data supports them;
- deterministic validation removes unsupported proposals;
- accepted KPI values are calculated locally;
- every KPI displays its source table, fields, formula, coverage, and evidence;
